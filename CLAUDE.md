# CLAUDE.md - 협업 규칙

이 레포에서 작업할 때 Claude Code(및 모든 협업자)가 반드시 지켜야 하는 Git 규칙입니다.

## 1. GitHub 협업 원칙 (필수, 예외 없음)

- **main 브랜치에 직접 커밋·푸시하지 않습니다.** 사소한 오타 수정이라도 예외 없음.
- 모든 변경 작업은 **새 브랜치를 만들어서** 진행하고, **PR(Pull Request)을 올려서** 머지합니다.
- 새 브랜치는 항상 `origin/main` 기준으로 생성합니다.
  ```
  git checkout -b <branch-name> origin/main
  ```
- 작업 시작 전 반드시 `git branch --show-current`로 현재 브랜치를 확인합니다.
  여러 작업(다른 탭, 다른 팀원)이 동시에 진행 중일 수 있으므로, 엉뚱한 브랜치에서 작업을 시작하지 않도록 주의합니다.
- PR 생성 전 `git log --oneline origin/main..HEAD`로 의도한 커밋만 포함되어 있는지 확인합니다.
- 무관한 변경사항(다른 작업의 미커밋 파일 등)은 `git add`로 함께 올리지 않습니다. 의도한 파일만 명시적으로 스테이징합니다.

## 2. 브랜치 이름 규칙

`<type>/<설명>` 형태로 짓습니다.

| type | 용도 | 예시 |
|---|---|---|
| `feature/` | 새 기능 | `feature/typhoon-map` |
| `fix/` | 버그 수정 | `fix/api-key-leak` |
| `docs/` | 가이드·문서 수정 | `docs/repo-select-screenshot` |

## 3. PR 워크플로우

1. `git checkout -b <branch> origin/main`
2. 작업 및 커밋
3. `git push -u origin <branch>`
4. `gh pr create`로 PR 생성 (제목·본문에 변경 이유를 명확히 작성)
5. 리뷰·머지는 팀원 또는 사용자 확인 후 진행 (Claude Code가 임의로 머지하지 않음)

## 4. AI를 활용한 PR 내용 작성 가이드

PR 제목과 본문을 잘 쓰면 리뷰어가 맥락을 빠르게 파악하고, 머지 후에도 변경 이력을 추적하기 쉬워집니다. Claude Code(또는 다른 AI 도구)에 아래 방식으로 요청하면 일관성 있는 PR 내용을 생성할 수 있습니다.

### 4-1. AI에게 PR 내용 생성 요청하기

작업이 끝난 뒤 Claude Code에게 다음과 같이 요청합니다.

```
이 브랜치의 변경 내용을 바탕으로 PR 제목과 본문을 작성해줘.
```

AI는 `git diff origin/main..HEAD`와 커밋 메시지를 분석해 아래 항목을 채워 줍니다.

### 4-2. PR 제목 규칙

- **70자 이내**로 작성합니다.
- `[타입] 한 줄 요약` 형식을 권장합니다.
  - `[feat]` 새 기능, `[fix]` 버그 수정, `[docs]` 문서, `[refactor]` 리팩터링, `[chore]` 기타

예시:

```
[docs] PR 작성 가이드 CLAUDE.md에 추가
[feat] 태풍 경로 지도 컴포넌트 구현
[fix] API 키 환경변수 누출 수정
```

### 4-3. PR 본문 템플릿

AI에게 아래 구조로 작성해 달라고 요청하거나, 직접 채워 넣습니다.

```markdown
## 변경 이유 (Why)
<!-- 왜 이 변경이 필요한가? 해결하려는 문제나 배경을 설명합니다. -->

## 변경 내용 (What)
<!-- 어떤 파일·기능을 어떻게 바꿨는지 핵심만 요약합니다. -->
- 

## 테스트 방법 (How to test)
<!-- 리뷰어가 직접 확인할 수 있는 단계를 적습니다. -->
1. 

## 참고 사항
<!-- 관련 이슈 번호, 스크린샷, 주의사항 등 -->
```

### 4-4. AI 프롬프트 예시

**기본 생성:**

```
현재 브랜치의 변경 내용을 분석해서 위 PR 본문 템플릿에 맞게 작성해줘. 한국어로.
```

**스크린샷이 있는 경우:**

```
이 스크린샷을 참고해서 UI 변경 사항을 PR 본문에 포함시켜줘.
```

**이슈와 연결:**

```
이 PR이 #<이슈번호>를 해결한다는 내용을 본문에 포함하고, Closes #<이슈번호> 키워드도 추가해줘.
```

### 4-5. 주의사항

- AI가 생성한 내용은 반드시 **직접 검토**하고, 부정확한 내용은 수정합니다.
- 민감한 정보(API 키, 개인정보 등)가 포함되지 않도록 확인합니다.
- PR 본문에 `Closes #이슈번호`를 넣으면 머지 시 해당 이슈가 자동으로 닫힙니다.

## 5. 예외 처리

예외는 없습니다. 사용자가 "이번엔 main에 바로 올려도 돼"처럼 **명시적으로** 요청한 경우에만 이 규칙을 건너뛸 수 있습니다. 명시적 지시가 없으면 항상 브랜치 + PR로 진행합니다.
